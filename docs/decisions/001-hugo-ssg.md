# ADR 001: Hugo를 정적 사이트 생성기로 선택

**상태:** 채택됨  
**날짜:** 초기 설정 시점

## 맥락

개인 블로그를 위한 SSG가 필요했다. 주요 후보: Hugo, Jekyll, Eleventy, Next.js.

## 결정

**Hugo**를 선택했다.

## 이유

- **빌드 속도**: 수백 개 포스트도 수초 내 빌드 완료. 콘텐츠가 늘어도 CI/CD 시간이 선형적으로 증가하지 않는다.
- **의존성 없음**: Go 단일 바이너리. Node.js/Ruby 없이도 `hugo` 명령 하나로 작동.
- **Page Bundle 구조**: `content/posts/{entry_id}/` 형태로 포스트와 이미지를 함께 관리. 이미지 처리(`images/`) 내장.
- **Go 템플릿**: 강한 타입, nil 안전성. 잘못된 변수 참조 시 빌드 타임에 오류 발생.
- **Permalinks 완전 제어**: URL 구조를 `hugo.yaml` 한 곳에서 명시적으로 선언. 우발적 변경 방지.

## 트레이드오프

- Go 템플릿 문법의 학습 곡선
- 플러그인 생태계는 Jekyll/Eleventy보다 작음
- 실시간 동적 기능(댓글, 좋아요)은 외부 서비스(Supabase) 필요

## 영향

- 콘텐츠는 반드시 Hugo가 인식하는 front matter 형식을 따라야 함
- 배포 산출물은 `public/` (빌드 후 정적 파일)
- 검색 기능은 빌드 후 Pagefind 인덱싱으로 구현
